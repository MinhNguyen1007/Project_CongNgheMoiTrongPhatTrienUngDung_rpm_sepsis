# 2.10. Thiết kế bộ test

Hệ thống giám sát sepsis mang tính **an toàn lâm sàng cao** — một lỗi phần mềm có thể dẫn tới bỏ sót cảnh báo hoặc cảnh báo sai gây mệt mỏi cho đội ngũ ICU. Do đó chiến lược test phải bao phủ đa tầng: từ đơn vị hàm nhỏ tới hệ thống đầu-cuối, từ hiệu năng dưới tải cao tới tính đúng đắn lâm sàng của mô hình.

## 2.10.1. Chiến lược test tổng thể

Áp dụng **mô hình kim tự tháp test** (test pyramid) với 4 tầng:

```
                    ▲
                   ╱ ╲    Model test (10%)
                  ╱───╲
                 ╱     ╲   Integration test (20%)
                ╱───────╲
               ╱         ╲  Load/performance test (10%)
              ╱───────────╲
             ╱             ╲ Unit test (60%)
            └───────────────┘
```

- **Unit test (60%)**: nhanh, chạy mỗi commit. Phủ các pure function (feature engineering, data transforms, scoring).
- **Integration test (20%)**: chậm hơn, chạy trên nightly CI. Phủ các luồng liên thành phần (Kinesis → Lambda → DB).
- **Load/performance test (10%)**: tuần 1 lần hoặc trước release. Đo latency và throughput end-to-end.
- **Model test (10%)**: sau mỗi lần train model mới. So sánh metric với baseline, kiểm tra fairness và calibration.

Các công cụ chính: `pytest` (Python), `vitest` (React), `Locust` (load), `testcontainers` (tạo LocalStack/PostgreSQL trong test), và `great_expectations` (data quality).

## 2.10.2. Unit test

### a) Test hàm feature engineering

Đây là nhóm test quan trọng nhất — sai lệch tính toán feature ảnh hưởng trực tiếp đến dự đoán. Ví dụ:

```python
def test_rolling_mean_6h():
    df = pd.DataFrame({
        "iculos": [1, 2, 3, 4, 5, 6, 7],
        "hr":     [80, 82, 84, 86, 88, 90, 92],
    })
    feat = extract_features(df, t=7)
    assert feat["hr_mean_6h"] == pytest.approx(87.0, abs=1e-6)
    assert feat["hr_min_6h"] == 82
    assert feat["hr_max_6h"] == 92

def test_qsofa_score():
    row = pd.Series({"resp": 24, "sbp": 95, "gcs": 14})
    assert compute_qsofa(row) == 3

def test_feature_with_all_nan():
    """Không được crash khi toàn bộ cửa sổ là NaN."""
    df = pd.DataFrame({"iculos": [1, 2, 3], "hr": [np.nan]*3})
    feat = extract_features(df, t=3)
    assert np.isnan(feat["hr_mean_6h"])
    assert feat["hr_missing_6h"] == 1.0
```

**Tiêu chí phủ**: mỗi hàm feature phải có ít nhất ba test — trường hợp bình thường, trường hợp biên (empty/single-point), và trường hợp toàn missing.

## 2.10.3. Integration test

### a) Luồng end-to-end streaming

Test tình huống "simulator → Kinesis (LocalStack) → Lambda → DynamoDB + PostgreSQL → API":

```python
@pytest.fixture(scope="session")

    with LocalStackContainer(services=["kinesis", "dynamodb", "s3"]) as ls:
        init_kinesis_stream(ls)
        init_dynamodb_tables(ls)
        yield ls

def test_e2e_vital_ingestion(localstack_services, pg_db):
    # Gửi 24h dữ liệu của 1 bệnh nhân
    producer = SimulatorProducer(endpoint=localstack_services.url)
    producer.send_patient_records("P999", hours=24)

    # Chờ consumer xử lý (timeout 30s)
    await_until(lambda: query_features("P999") is not None, timeout=30)

    features = query_features("P999")
    assert features["iculos_hours"] == 24
    assert "hr_mean_6h" in features
    assert 60 <= features["hr_mean_6h"] <= 140

def test_e2e_alert_triggered_for_sepsis_case(localstack_services, pg_db):
    """Phát lại một ca sepsis thật từ test set → hệ thống phải sinh alert."""
    producer = SimulatorProducer(endpoint=localstack_services.url)
    producer.replay_patient_from_dataset(patient_id="p021548", speed="fast")

    await_until(
        lambda: pg_db.query("SELECT * FROM alerts WHERE patient_id='p021548'").first(),
        timeout=60,
    )
    alerts = pg_db.query("SELECT * FROM alerts WHERE patient_id='p021548'").all()
    assert len(alerts) >= 1
    assert alerts[0].severity in ("WARNING", "CRITICAL")
```

## 2.10.4. Load test và performance test

### a) Mục tiêu hiệu năng

| Chỉ số                         | Mục tiêu           | Ghi chú                                      |
| ------------------------------ | ------------------ | -------------------------------------------- |
| Latency p50 inference          | < 100ms            | Từ event Kinesis đến có prediction           |
| Latency p95 inference          | < 500ms            | SLA chính                                    |
| Latency p99 inference          | < 1000ms           |                                              |
| Throughput                     | ≥ 100 BN đồng thời | = 100 event/giây (1 event/BN/giờ giả lập 1s) |
| WebSocket broadcast delay      | < 200ms            | Từ prediction đến browser                    |
| API response p95 (GET patient) | < 300ms            |                                              |

### b) Kịch bản Locust

```python
class SepsisMonitoringUser(HttpUser):
    wait_time = between(1, 3)

    @task(3)
    def get_patient_list(self):
        self.client.get("/api/patients?status=active")

    @task(2)

        pid = random.choice(self.patient_ids)

                         json={"decision": "monitoring", "note": "load test"})
```

Chạy: `locust -f tests/load/locustfile.py --users 100 --spawn-rate 10 --run-time 10m`.

### c) Chaos / stress test

Kiểm thử các kịch bản lỗi:

- **Kinesis tạm ngắt 30 giây**: consumer phải retry, không mất event, báo cáo catch-up lag.
- **Model inference crash**: API phải trả 503 với `Retry-After` header, không treo WebSocket client.
- **Burst traffic 10x**: hệ thống queue backpressure, không drop event sinh tồn.
- **DynamoDB throttling**: exponential backoff hoạt động, fallback sang đọc S3.

## 2.10.5. Model test

Nhóm test riêng cho mô hình ML, chạy sau mỗi MLflow run `Staging → Production`:

### a) Metric gate

Model mới chỉ được promote nếu đạt toàn bộ ngưỡng sau trên test holdout:

```python
def test_model_metric_gates(candidate_model, test_set):
    preds = candidate_model.predict_proba(test_set.X)[:, 1]
    assert utility_score(test_set.y, preds) >= 0.35
    assert roc_auc_score(test_set.y, preds) >= 0.80
    assert sensitivity_at_specificity(test_set.y, preds, spec=0.95) >= 0.40
```

### b) Regression test

Dùng một bộ "golden test cases" — các ca lâm sàng đặc biệt (septic shock sớm, pneumonia → sepsis, false-positive cũ đã fix). Model mới phải xử lý đúng các case này:

```python
GOLDEN_CASES = load_golden_cases("tests/model/golden/")

@pytest.mark.parametrize("case", GOLDEN_CASES)
def test_golden_case(case, candidate_model):
    prob = candidate_model.predict_proba(case.features)[0, 1]
    if case.expected == "high_risk":
        assert prob >= 0.5, f"{case.id}: expected alert but prob={prob:.2f}"
    else:
        assert prob < 0.5
```

### c) Fairness test

So sánh sensitivity và specificity giữa các sub-group:

```python
def test_fairness_across_age_groups(candidate_model, test_set):
    for group in ["<50", "50-70", ">70"]:
        subset = test_set[test_set.age_group == group]
        sens = recall_score(subset.y, candidate_model.predict(subset.X))
        assert sens >= 0.60, f"Sensitivity nhóm {group} quá thấp: {sens:.2f}"
    # Chênh lệch giữa nhóm cao nhất và thấp nhất không quá 15%
    assert max(sensitivities) - min(sensitivities) <= 0.15
```

### d) Calibration test

Kiểm tra probability calibration bằng Brier score và reliability diagram:

```python
def test_calibration_brier(candidate_model, test_set):
    probs = candidate_model.predict_proba(test_set.X)[:, 1]
    brier = brier_score_loss(test_set.y, probs)
    assert brier <= 0.08  # càng nhỏ càng tốt
```

## 2.10.7. Security test

- **Bandit**: scan Python code cho lỗ hổng phổ biến (hardcoded secret, unsafe pickle, SQL injection).
- **Gitleaks**: pre-commit hook quét secret trước khi commit.
- **Dependabot** (GitHub): auto-PR cập nhật dependency có CVE.
- **Auth test**: JWT expiry, role-based access (NURSE không được sửa threshold), rate limiting API.
- **Input sanitization**: test API chống injection (`' OR 1=1 --`, XSS payload trong note field).

## 2.10.8. Test data và test fixtures

### a) Tập dữ liệu test

Ba tập riêng biệt:

1. **Synthetic fixtures** (committed vào repo): 10 bệnh nhân giả định nhỏ cho unit test, file `tests/fixtures/synthetic_patients.json`.
2. **Sample dataset** (DVC hoặc LocalStack S3): 100 bệnh nhân thật từ PhysioNet, cho integration test.
3. **Holdout set** (không bao giờ chạm trong dev): 15% dataset, chỉ dùng báo cáo model metric cuối cùng.

### b) Sinh dữ liệu ngẫu nhiên có cấu trúc

Dùng `hypothesis` (property-based testing) cho các hàm feature:

```python
from hypothesis import given, strategies as st

@given(
    hrs=st.lists(st.floats(min_value=40, max_value=200),
                 min_size=1, max_size=24)
)
def test_rolling_mean_is_bounded_by_min_max(hrs):
    df = pd.DataFrame({"iculos": range(len(hrs)), "hr": hrs})
    feat = extract_features(df, t=len(hrs))
    assert feat["hr_min_6h"] <= feat["hr_mean_6h"] <= feat["hr_max_6h"]
```

Property-based test phát hiện edge case mà developer không nghĩ tới.

## 2.10.9. Tích hợp CI/CD

### a) Pipeline GitHub Actions

Hai workflow chính:

**`ci.yml`** (chạy mỗi PR):

```yaml
jobs:
  lint:         ruff + black + mypy + prettier
  unit:         pytest -m "not integration" --cov + vitest
  security:    bandit + gitleaks + npm audit
  build:       docker build (tag: pr-<sha>)
```

**`integration.yml`** (chạy nightly + trước merge vào main):

```yaml
jobs:
  integration:
    - startup localstack + postgres + mlflow
    - pytest -m integration
    - teardown
  model-gate:
    - load latest MLflow candidate
    - run model metric gate + fairness + golden cases
  load-test:
    - locust headless 10 phút 100 users
    - assert p95 < 500ms
```

### b) Quy tắc merge

- PR chỉ merge được khi toàn bộ CI xanh.
- Coverage không được giảm > 2% so với main.
- Ít nhất 1 reviewer approve.
- Với PR đổi ML model hoặc alert logic, bắt buộc thêm reviewer là role B (ML) hoặc bác sĩ cố vấn (nếu có).

## 2.10.10. Test acceptance cuối kỳ (demo checklist)

Trước buổi bảo vệ, nhóm chạy danh sách kiểm thử chấp nhận:

- [ ] `docker-compose up -d` → tất cả service healthy < 3 phút
- [ ] `pytest tests/` pass 100%, coverage ≥ 85%
- [ ] Simulator phát 10 BN → Dashboard hiện real-time < 5 giây lag
- [ ] Một ca sepsis thật → alert kích hoạt đúng thời điểm, SHAP hiển thị feature hợp lý
- [ ] Bác sĩ xác nhận alert → ghi vào audit_log, trạng thái cập nhật trên UI
- [ ] Admin thay đổi threshold → áp dụng cho prediction mới trong < 30s
- [ ] Load test Locust 100 users → p95 < 500ms, không lỗi 5xx
- [ ] Drift scenario (đổi phân phối) → Evidently báo cáo + Slack alert < 5 phút
- [ ] MLflow UI hiện ≥ 2 experiments với ≥ 5 runs, model registry có Production stage
- [ ] Grafana dashboard hiển thị latency + throughput timeline 24h qua
- [ ] CI pipeline GitHub Actions xanh trên main branch
